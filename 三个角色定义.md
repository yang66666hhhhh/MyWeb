这三个角色建议定义成“平台级权限层级”，不要和“身份 Persona”混在一起。

`Member`：普通成员  
主要是使用系统。能看自己的工作台、成长、工作日志、任务、习惯、知识库等基础功能。数据权限默认只看自己的数据。适合学生、教师、开发者、实施工程师这些真实用户身份的普通账号。

`Pro`：高级用户 / 专业用户  
比 Member 多一些高级功能和跨模块能力，比如统计看板、项目管理、设备管理、AI 总结、数据导入等。数据权限可以根据业务给到“团队/项目范围”，但不建议默认拥有平台配置权。Pro 更像“付费/高级能力层”，不是管理员。

`Owner`：平台所有者 / 平台管理者  
拥有系统配置和治理权限，比如用户管理、身份类型、角色菜单、功能开关、订阅/套餐、全局配置、审计和故障处理。Owner 可以看到和调整全局资源，也能访问所有功能。

平台管理者不应该“只负责管理”。更准确地说：

Owner 是最高权限账号，既能使用业务功能，也能管理平台。但日常产品设计上，Owner 的核心职责是平台治理；业务使用身份仍然通过 Persona 表达。比如同一个账号可以是：

- 角色：Owner
- 身份：开发者、实施工程师、教师

这样它既有平台管理权限，也能因为“教师身份”看到教学中心，因为“实施工程师身份”看到实施相关能力。

你现在这套模型可以理解为：

```text
Role 决定权限等级：Member / Pro / Owner
Plan 决定订阅能力：Free / Pro / Team
Persona 决定工作场景：教师 / 学生 / 开发者 / 实施工程师 / 设计师...
FeatureCode 决定具体功能入口和接口权限
```

当前实现里，`FeatureCode` 不是简单的 `PlanFeatures ∪ PersonaFeatures`。更准确的规则是：

```text
非 Persona 类功能 = 当前有效套餐包含即可
Persona 类功能 = 当前有效套餐包含 && 用户拥有对应启用 Persona
Owner = 全部启用功能
```

这样可以避免一个普通 Member 只因为有 Developer 或 Student 身份，就绕过套餐边界看到所有开发/学生高级能力。

前端体验也要跟这个模型一致：

- 菜单入口由后端 `/api/system/role-menus/mine` 统一过滤。
- 页面内部的统计卡、下拉数据、快捷入口、操作按钮也要按 `accessCodes` 再过滤。
- Member 不应该看到 Pro 专属入口，也不应该因为页面内部请求 Pro 接口而弹 403。
- 基础页面需要降级展示：没有高级功能码就隐藏高级区块，或者把高级字典下拉降级成普通输入。

所以推荐规则是：

- `Owner`：平台治理 + 全功能兜底
- `Pro`：高级功能使用者，不默认管理平台
- `Member`：基础功能使用者
- `Persona`：决定出现哪些“场景菜单”
- `FeatureCode`：决定能不能访问具体 API/页面

你问“平台管理者是不是就只负责管理”，答案是：不是。它是最高权限层，但最好把“管理能力”和“业务身份”分开表达。这样系统以后不会乱。
